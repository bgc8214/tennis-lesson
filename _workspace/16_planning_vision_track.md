## 1. 현황 평가 — "제대로 되고 있는가"

### 1-1. 보유 자산 인벤토리

| 자산 | 상태 | 평가 |

| `court_service.py` (코트 전술: 클립 다운로드 → YOLOv8 위치 추출 → Gemini 전술 생성) | 구현 완료, **기본 비활성** (`COURT_ANALYSIS_ENABLED=False`) | 아래 1-2 참조 — 현 상태로 켜면 안 됨 |

| 13문서 축2 (스윙 통계, 하이라이트 컷) | 기획만 | 착수 순서가 이 문서의 주제 |

### 1-2. court_service 판정: 지금 켜면 안 되는 3가지 이유

1. **깨진 입력 상속**: 프롬프트가 Phase 1 리포트의 timestamps/quote를 입력받고
   "위치 정보가 없으면 quote/label에서 추론"하도록 되어 있음. quote 정밀도가
   15.6%로 실측된 지금, 이 경로는 **오염된 입력 위에 생성형 추론을 얹는 구조**.
2. **검증 게이트 부재**: `tactic`(전술 행동 1문장)은 Gemini의 생성 결과인데
   Phase 1이 받은 코드 레벨 검증에 해당하는 장치가 없음 — 인사이트 ①(출처
   위장 금지) 기준으로 "AI 추정" 라벨 없이는 노출 불가.
3. **정확도 실측 이력 없음**: 6월 QA(07문서)는 인터페이스 정합성 검증이었지
   위치/전술의 **정확도** 검증이 아님. YOLO 9존 스냅이 실제 영상에서 맞는지
   골든셋급 실측이 없음.

**처분**: 삭제하지 않되(코드/UI 자산 유효) 기본 off 유지. 재활성 조건 =
3절 V-0 실측 통과 + tactic 문장의 "AI 추정" 라벨링. YOLO 위치 추출 부분은
V-1(스윙 통계)의 부품으로 재사용 가치 있음.

### 1-3. 종합: "제대로 되고 있는가" → 절반만

비전 트랙은 코드가 없어서가 아니라 **오디오 트랙이 밟은 함정(실측 없이 구현
→ 나중에 정밀도 붕괴 발견)을 그대로 밟기 직전 상태로 멈춰 있는 것**이 현재
상황이다. 다행히 꺼져 있어 사고는 없었다. 15문서의 교훈을 비전에 선제
적용하는 것이 이 문서의 핵심이다: **구현보다 실측이 먼저.**

## 2. 전략 판정 — "필요한가"

**결론: 필요하다. 인용 제품이 죽은 지금 오히려 더 필요해졌다. 단, coarse와
fine을 구분해야 한다.**

근거:
- 인용(텍스트) 가치가 축소된 지금, 서비스의 차별화 축은 모먼트 내비게이션
  + **시각 가치**로 재편됨 (13문서 배경 그대로). 원거리 촬영에서 목소리는
  죽지만 몸은 화면에 잘 나온다.
- PMD 수익 모델의 프로 패스(9,900원) 기둥이자 기술 진입장벽.
- 단, 오디오 트랙 실측이 준 교훈: **원거리에서 견고한 것(coarse)과 깨지는
  것(fine)을 먼저 구분하라.**

| 구분 | 항목 | 원거리 견고성 | 판정 |
|---|---|---|---|
| **Coarse** | 스윙 횟수, 포핸드/백핸드 비율, 랠리 길이, 활동량, 하이라이트 컷, 코트 존 수준 위치 | 높음 (사람 박스/큰 동작 감지는 해상도 요구 낮음) | **지금 트랙의 대상** — 정밀도 주장 없이 성립, 오디오 실패의 보완재 |
| **Fine** | 서브 타점 각도, 라켓드롭 깊이, 내전 각도 (PMD 5.2) | 낮음~미지수 (원거리+임의 앵글에서 관절 각도 신뢰도 불명) | **보류** — 근접/적절 앵글 영상 확보 + V-0급 실측 + 유료화 시점(12문서 Stage 3)에 재개 |

Fine을 서두르면 안 되는 이유는 quote와 동일하다: 검증 안 된 "내전 각도 23°"를
보여주는 것은 검증 안 된 인용을 보여주는 것과 같은 신뢰 자살이다.

## 3. 실행 계획 — 실측 우선 (15문서 교훈의 이식)

### V-0. 비전 골든셋 실측 — **3파전 비교** (구현 전 필수, 로컬 세션 적합)

오디오 골든셋과 동일한 방법론. 이미 보유한 골든셋 영상 3~4개로,
**세 후보를 같은 정답지에 대해 나란히 채점**한다 (스택 결정을 데이터로):

| 후보 | 비고 |
|---|---|
| MediaPipe Pose (BlazePose) | 브라우저 지원 최상. 단 근거리·1인 학습 모델이라 원거리 소인물/다인(게임레슨)에서 깨질 위험 — 정확히 이걸 재는 것 |
| YOLOv8-pose | 멀티퍼슨+소인물에 더 견고 예상, 서버엔 ultralytics 기설치, ONNX로 브라우저(onnxruntime-web) 실행도 가능 |
| **타구음 온셋 감지 (비전 아님)** | 공 임팩트는 짧고 강한 타악 이벤트라 원거리 마이크에서도 명확 — 발화 STT가 죽은 것과 무관하게 견고. librosa 등 고전 DSP로 스윙(타구) 횟수/랠리 리듬/활동 구간 추출. **기존 오디오 파이프라인 재사용 = 최저 비용.** 한계: 포핸드/백핸드 구분·하이라이트 컷 불가 |

절차:
1. 한 영상에서 스윙(타구) 50~100개를 손으로 라벨(타임스탬프+종류) →
   `backend/tests/golden_vision/` 정답지로 저장
2. 프레임 샘플링(예: 2fps) → 두 포즈 모델의 인물/포즈 검출률 측정
3. 세 후보 각각으로 자동 카운트 → 정답 대비 P/R 채점 (`eval_golden_vision.py`)

**게이트**: P/R 목표치(제안: 각 80%) — 오디오 트랙처럼 "만들고 나서 15%임을
발견"하는 일을 반복하지 않는다. **타구음이 게이트를 통과하면 V-1의 카운트류
지표는 비전 없이 타구음으로 출시하고, 비전은 구분(포핸드/백핸드)·하이라이트
컷에만 투입하는 하이브리드가 기본 안이 된다.**

### V-1. 스윙 통계 "테니스 인바디" + 하이라이트 컷 (V-0 통과 시)

- 리포트에 오디오 무관 섹션 추가: 오늘 스윙 N회(포핸드/백핸드 비율),
  활동 시간, 하이라이트 구간 상위 3개(YouTube `?t=` 링크 — 다운로드 불필요)
- 모든 수치에 산출 방식 표기(예: "영상 기반 자동 집계, 오차 있을 수 있음") —
  2계층 라벨링 원칙의 비전 버전
- 실행 위치: 도그푸딩 단계에서는 로컬(worker) 실행 권장 — 영상 다운로드가
  필요하므로 서버(봇감지+트래픽 비용) 부적합 (아래 4절)

### V-2. 온디바이스 비전 PoC — 장기 아키텍처의 핵심 결정

**문제**: Phase 3 직접 업로드는 "오디오만" 서버로 보내는 설계(11문서) —
비전은 영상 프레임이 필요해 정면 충돌한다. 서버가 영상을 받으면 무저장
원칙·비용 0원 구조·초상권 리스크 해소가 전부 무너진다.

**해법**: MediaPipe는 브라우저(WASM/WebGL)에서 실행 가능하다.
FFmpeg.wasm으로 오디오를 추출하는 것과 정확히 같은 철학으로, **포즈/스윙
감지도 유저 브라우저에서 수행하고 서버에는 랜드마크·이벤트 JSON(수십 KB)만
업로드**한다.

| | 서버 사이드 비전 | 온디바이스 비전 |
|---|---|---|
| 영상 전송 | 수백 MB/회 (봇감지 또는 업로드 비용) | **0 — 영상이 기기를 떠나지 않음** |
| 서버 비용 | GPU/CPU 상당 | JSON 수신뿐 (~0원) |
| 초상권/개인정보 | 서버가 영상 보유 | **원천 해소 — 마케팅 포인트** |
| 처리 시간 | 서버 큐 | 유저 기기 성능 의존 (저사양 폰 리스크) |

**[2026-07-20 정정] 온디바이스는 "유일해"가 아니라 "후보 중 하나"다.**
재검토 결과 아래 리스크와 대안을 반영해 강등한다:

- **리스크 3개**: ① 모바일 연산량 — 60분 영상 2fps 샘플링만 해도 7,200프레임,
  폰 브라우저에서 10~20분 + 발열 + 화면 유지 필요. "폰 붙들고 15분 대기"는
  셀프 음성 메모를 강등시킨 것과 같은 행동경제학 문제. ② BlazePose는
  근거리·1인 학습 모델 — 원거리 소인물 검출력 미지수 (V-0가 판정).
  ③ 다인(게임레슨) 미지원 — 검출+트래킹 추가 필요.
- **서버 사이드 재평가**: 4절의 "회당 $0.5~1"은 yt-dlp+residential 프록시로
  영상을 *다운로드*하는 경우다. **유저가 영상을 직접 업로드하면 ingress는
  무료**이고, 서버리스 GPU(초 단위 과금, T4급) 처리비는 회당 수십 원 수준 —
  즉시 삭제 정책으로 무저장 원칙도 유지 가능. 진짜 트레이드오프는
  "유저의 1GB 업로드 부담(Wi-Fi 전제) vs 유저 기기 연산 부담"이다.
- 따라서 V-2 PoC의 판정 질문을 확장한다: (a) 브라우저 포즈 추정이 폰에서
  현실적인가, (b) 아니라면 직접 업로드 + 서버리스 GPU 경로의 업로드 UX가
  수용 가능한가 — 둘 중 이기는 쪽이 장기 아키텍처.

### V-3. Fine 관절 분석 (보류 — 재개 조건 명시)

재개 조건 3가지가 모두 충족될 때만: ① 근접/측면 촬영 영상 확보(촬영 가이드
준수 유저 데이터), ② V-0 방법론으로 각도 측정 오차 실측 통과, ③ 유료화
단계 진입(12문서 Stage 3, 프로 패스 판매 시점). 그 전에는 어떤 각도 수치도
사용자에게 노출하지 않는다.

## 4. 제약 사항 (전 단계 공통)

- **영상 다운로드는 오디오보다 비싸다**: 480p 60분 ≈ 수백 MB — residential
  프록시 경유 시 회당 $0.5~1로 오디오($0.05)의 10~20배. 서버 사이드 비전은
  스케일에서 성립하지 않음 → V-2(온디바이스)가 장기 정답인 이유.
- **도그푸딩 단계 실행 위치는 로컬**: 로컬 yt-dlp 정상 + GPU/CPU 자유.
  V-0/V-1은 전부 로컬에서 수행 가능.
- Cloud Run에는 현재 ultralytics/opencv가 requirements에 포함되어 이미지가
  무겁다 — 비전을 로컬/온디바이스로 확정하면 서버 이미지에서 분리(경량화)
  검토 여지.

## 5. 우선순위 요약

```
V-0 비전 골든셋 실측 (로컬, 구현 아님 — 지금 착수 가능, 베타 준비와 병행 무방)
  ↓ 게이트: 스윙 감지 P/R ≥ 80%
V-1 스윙 통계 + 하이라이트 컷 (13문서 축2, 로컬 실행)
  ∥ 병행
V-2 온디바이스 PoC (Phase 3 아키텍처 결정용)
  ↓ 조건 충족 시에만
V-3 Fine 관절 분석 (근접 촬영 + 실측 통과 + Stage 3)
```

베타 준비(👍/👎·배포·약관 트랙)가 여전히 최우선이며, 비전은 V-0(실측)만
병행 착수를 권장 — 구현 리소스를 먹지 않는 측정 작업이기 때문.
