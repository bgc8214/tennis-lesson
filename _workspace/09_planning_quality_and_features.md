## 1. 전사/구조화 품질 개선 — 우선순위순 태스크

각 항목은 독립적으로 구현 가능하도록 작성함. 담당 에이전트는 backend 기본,
④는 UI 병행 필요.

### 1-1. [즉시 착수 권장] Whisper `initial_prompt`에 테니스 용어 사전 주입

- **현황**: `backend/app/services/stt_providers.py`의 `_transcribe_local()`,
  `_groq_transcribe_file()` 모두 `initial_prompt` 미사용 확인됨.
- **작업**: `model.transcribe(..., initial_prompt=TENNIS_TERM_HINT)` 및
  Groq API 호출의 `prompt` 파라미터에 동일 문자열 전달.
  예시: "라켓드롭, 임팩트, 스플릿스텝, 내전, 팔로우스루, 하프발리, 크로스,
  다운더라인, 타점, 트로피자세, 토스, 발리, 로브" 등 실제 프롬프트에 이미
  등장하는 용어 목록(`gemini_service.py`의 `ALLOWED_LESSON_TYPES` 및
  청크 프롬프트 하단 "테니스 용어 참고" 문구 재사용 가능).
- **기대 효과**: 도메인 용어 인식률 상승. 코드 변경량 적음, 리스크 거의 없음.
- **검증**: 1-3의 골든셋으로 before/after WER(용어 단위) 비교.

### 1-2. [우선순위 최상] STT 전처리: 라우드니스 정규화 + 하이패스 필터

- **현황**: `_download_full_audio()`/`_reencode_for_upload()` 어디에도
  `loudnorm`/`highpass`/`afftdn` 미적용 확인됨 (원본 볼륨 그대로 STT 입력).
- **문제 근거**: 사용자 실측 — mean_volume -27~-40dB (표준 발화 대비 훨씬 낮음),
  이 때문에 no_speech_prob가 높게 나와 실제 발화가 사후 필터에서 유실될
  가능성이 있음 (517개 중 186개 no_speech 폐기, 이 중 일부는 진짜 발화로 추정).
- **작업**: `backend/app/services/stt_providers.py`의 오디오 준비 단계(로컬 경로는
  `_transcribe_local` 진입 전, Groq 경로는 `_reencode_for_upload()`)에
  ffmpeg 필터 체인 추가:
  ```
  ffmpeg -i in.wav -af "highpass=f=100,loudnorm=I=-16:TP=-1.5:LRA=11" out.wav
  ```
  - `highpass=f=100`: 코트 저주파 잔향/바람 노이즈 제거
  - `loudnorm`: EBU R128 라우드니스 정규화 (-16 LUFS 표준 발화 타깃)
- **리스크**: 노이즈도 함께 증폭될 수 있음 → 반드시 1-3 골든셋으로 A/B 후 채택.
  compression_ratio 필터(환청 감지)가 증폭된 노이즈에 오탐하지 않는지 함께 확인.

### 1-3. [전제 조건] 골든셋 회귀 스크립트 — 다른 개선 항목보다 먼저 구축 권장

- **작업**: 이미 검증된 영상 2개(`5PUGx-OYI5s`, `4mr0tVIu9sw`)의 코치 발언을
  손으로 정답지(JSON, 발언+타임스탬프)로 작성 → `backend/tests/golden/` 등에 저장.
  파이프라인 실행 결과와 비교해 재현율(정답 발언 중 몇 %가 검증 통과했는지)을
  자동 채점하는 스크립트 작성 (`backend/scripts/eval_golden.py` 제안).
- **이유**: 1-1, 1-2, 1-4, 1-5 모두 "감으로 나아졌다"가 아니라 이 스크립트
  점수로 판정해야 함. 프롬프트/전처리 변경이 누적되면 회귀 없이는 품질
  방향을 잃기 쉬움.
- **담당**: backend 에이전트, QA 에이전트와 공동 작업 적합.

### 1-4. [서비스화 전 필수] 하드코딩된 화자 가정 제거

- **현황**: `backend/app/services/gemini_service.py`의 7개 프롬프트
  (`CHUNK_PROMPT`, `MERGE_PROMPT_TEMPLATE`, `TRANSCRIPT_PROMPT`,
  `YOUTUBE_URL_REPORT_PROMPT`, `YOUTUBE_URL_SEGMENT_PROMPT_TEMPLATE(_UNKNOWN_DURATION)`,
  `WHISPER_TRANSCRIPT_PROMPT`(구 미사용), `WHISPER_VERIFIED_PROMPT`)에
  "여성 코치가 남성 수강생에게"가 하드코딩되어 있음 (grep 확인, 총 8곳).
- **문제**: 사용자 도그푸딩 데이터엔 맞지만, 코치/수강생 성별 조합이 다른
  영상이 들어오면 화자 구분이 반대로 뒤집혀 **수강생 발화가 코치 피드백으로
  오인**될 위험. 서비스 오픈 전 반드시 해소해야 하는 정확성 이슈.
- **작업 (Phase A — 최소 변경)**: 레슨 등록 시 선택 필드
  (`coach_label`, 예: "코치"/"쌤" + 대명사 없는 중립 표현으로 프롬프트 전면 수정,
  예: "코치가 수강생에게 진행한 1:1 레슨"으로 성별 언급 자체를 제거)로
  전환. DB 스키마 변경 불필요, 프롬프트 텍스트만 수정.
- **작업 (Phase B — 장기, 선택)**: 전사 단계에서 화자 분리(diarization) 후
  발화량/질문형 패턴으로 코치/수강생 자동 추정. Whisper 자체는 diarization
  미지원이라 별도 모델(pyannote 등) 필요 — 비용/복잡도 대비 효과를 먼저 검토.
- **권장**: Phase A만 지금 하고 Phase B는 백로그로.

### 1-5. Gemini 구조화 2단 분리 (추출 vs 코칭 종합)

- **현황**: `generate_lesson_report_whisper()`가 `WHISPER_VERIFIED_PROMPT` 단일
  호출로 추출+카드 요약을 동시에 수행.
- **작업**: 함수를 2단 호출로 분리
  - **Pass A (추출·분류)**: 전사 텍스트 → 코치 발언만 교정/드릴/전술 분류 +
    카테고리별 반복 횟수 집계, temperature=0.0 (기존 timestamps 추출 부분과 동일 계약)
  - **Pass B (코칭 종합)**: Pass A의 **검증 게이트 통과 후** 결과만 입력받아
    card1/2/3 + steps/scenarios 생성. "가장 반복된 지적"을 정량 근거로 명시하도록
    프롬프트에 반복 횟수 포함.
  - `verification.verify_report()`는 Pass A 출력에 적용 (지금과 동일 지점),
    Pass B는 이미 검증된 사실만 보므로 별도 검증 불필요.
- **효과**: card1(고질병)이 "3회 반복 지적"처럼 정량 근거를 갖게 됨.
  API 응답 shape 변경 없음(내부 구현만 분리).

### 1-6. "검증된 사실" vs "AI 보조 설명" 2계층 리포트 — 이번 기획 핵심 제안

- **배경**: 현재 프롬프트는 "테니스 지식으로 채우지 마라"를 극단적으로 강제해
  안전하지만 건조함. 문제의 본질은 창작이 아니라 **출처 위장**이므로,
  출처를 명확히 라벨링하면 AI의 일반 지식을 안전하게 활용 가능.
- **작업**:
  - 백엔드: `verification.verify_report()` 통과 항목은 기존대로 `timestamps[]`,
    card 필드에 유지. 별도로 `coach_notes[]` 또는 `ai_context` 필드 신설 —
    검증 게이트를 거치지 않는(=통과 실패해도 되는) AI 일반 설명
    (용어 정의, 왜 이 교정이 중요한지, 추천 셀프 드릴). 이 필드는 quote 필드
    자체가 없으므로 verification 대상에서 애초에 제외.
  - 프론트: `frontend/src/components/ReportView/NoteCards.tsx`,
    `FeedbackTimeline.tsx`에 "🎾 코치님 말씀"(기존, match_score 뱃지 노출)과
    "💡 AI 코치 노트"(신규, 별도 스타일 + "AI 보조 설명" 라벨 고정 노출) 시각적 구분.
- **주의**: 라벨링이 흐려지면 안 됨 — AI 노트 영역은 코치 인용 영역과 시각적으로
  명확히 분리(배경색/아이콘 등)되어야 사용자가 혼동하지 않음. QA 체크리스트에
  "AI 노트에 quote/코치 발언 형태 문구가 섞이지 않는지" 항목 추가 필요.

### 1-7. 레슨 간 연속성 (코치의 기억력) 주입

- **작업**: `_run_analysis_pipeline()`(`backend/app/routers/lessons.py`)에서
  Pass B(1-5 참조) 호출 전, 같은 `user_id`의 직전 1~3개 `lesson_reports`
  (`card1_problem`, `keywords`)를 Supabase에서 조회해 프롬프트 컨텍스트로 주입.
  예: "지난 레슨 지적: [왼손 하강, 라켓드롭 부족] — 이번 레슨에도 반복되면
  card1에 '지난 N회 연속 지적'으로 명시."
- **효과**: PMD 비전("성장 히스토리")과 직결, 기존 DB 스키마로 구현 가능
  (신규 테이블 불필요).
- **의존성**: 1-5(2단 분리) 이후 Pass B 시점에 넣는 게 자연스러움.

### 1-8. 구조화 모델 상향

- **현황**: `backend/app/config.py` `GEMINI_MODEL` 기본값 `gemini-1.5-flash`(구세대).
- **작업**: 최소한 Pass B(코칭 종합, 1-5 참조)만이라도 상위 모델로 전환 검토.
  전사 텍스트만 입력이라 토큰량이 적어(오디오 직접 입력 대비) 비용 영향 작음.
- **우선순위**: 낮음 — 1-1/1-2/1-4 이후 골든셋 점수로 필요성 재판단.

## 3. 구현 위임 시 참고

- 품질 개선(1절)은 backend 에이전트 단독 작업 가능한 항목이 대부분
  (1-6만 frontend 병행).
- 기능 확장(2절)은 착수 항목별로 architect → backend/frontend 순으로
  이 문서의 tennis-orchestrator 하이브리드 모드를 따를 것.
- 각 항목 착수 시 이 문서의 해당 섹션 번호(예: "1-2 착수")를 명시하면
  구현 에이전트가 이 파일에서 근거/파일경로를 그대로 가져다 쓸 수 있음.
