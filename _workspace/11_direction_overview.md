2. **Cloud Run 봇 감지 미해결** — 코드 준비는 끝났으나 실제 우회 수단(프록시 또는 구조 변경)은 미가동
3. **서비스화 비용 구조** — 무료/유료 경계를 어디에 둘 것인가

## 2. 핵심 인사이트 3가지 (모든 의사결정의 기준)

### 인사이트 ① 할루시네이션의 본질은 "창작"이 아니라 "출처 위장"이다

LLM이 테니스 지식을 말하는 것 자체는 문제가 아니다. **코치가 한 말인 척하는 것**이 문제다.
→ 따라서 해법은 금지가 아니라 **출처 분리**: 검증 게이트를 통과한 "코치님 말씀"과
명시 라벨이 붙은 "AI 코치 노트"를 시각적으로 분리하면, 안전과 풍부함을 동시에 얻는다
(09문서 1-6). 검증 게이트는 이미 있으므로 남은 것은 2계층 표시 구현뿐.

### 인사이트 ② 검증 게이트는 "전사 충실성"을 보증하지 "현실 충실성"을 보증하지 않는다

Whisper가 잘못 들은 문장은 검증을 통과한다 — 쓰레기 전사가 들어가면 검증된
쓰레기가 나온다. 60분 실전 케이스(3절)가 이를 실증했다.
→ 품질 투자의 최우선 순위는 프롬프트가 아니라 **전사 정확도**다:
전처리(loudnorm/highpass) + large-v3 + 테니스 용어/참가자 이름 initial_prompt
(09문서 1-1/1-2, 10문서 local-max). 그리고 개선 효과는 감이 아니라
**골든셋 회귀 점수**(09문서 1-3)로 판정한다.

### 인사이트 ③ 봇 감지의 본질은 코드가 아니라 "IP가 어디서 오는가"다

프롬프트/파이프라인을 아무리 고쳐도 Cloud Run의 데이터센터 IP는 차단당한다.
업계 공통 문제이며 해법은 정확히 세 갈래뿐:

| 해법 | 비용 | 본질 |
|---|---|---|
| residential proxy | ~$0.05/회 | 돈으로 주거용 IP를 빌림 |
| 로컬 worker (08문서 b안) | 0원 | 다운로드를 주거용 IP 머신(집)으로 옮김 |
| **직접 업로드 (Phase 3)** | **0원** | **YouTube에서 받는 행위 자체를 제거** — 근본 해결 |

## 3. 실전 케이스 스터디 — 60분 게임 레슨 (aYA3iILW2B0)

whisper 검증 경로로 실제 60분 영상을 처리한 결과 (2026-07-17 사용자 실측):
카드 8개 생성, 인용 다수가 STT 오인식으로 깨짐, 앞 20분 공백.

| 증상 | 원인 진단 | 대응 (문서 위치) |
|---|---|---|
| 인용문 깨짐 ("수리발이", "가발이 뛰어요") | STT 오인식 — 검증 게이트는 정상 동작 (인사이트 ②) | 전사 상향: 09문서 1-1/1-2 + 10문서 local-max |
| 60분에 8개, 앞 20분 공백 | 퍼널 4단계(STT필터→60분 통짜 Gemini 호출→검증→압축) 중 손실 지점 미계측. 긴 컨텍스트에서 LLM 과소 추출 경향 + "15~30개" 고정 제한 | ① 로그의 필터/검증 통계로 손실 지점 확인 ② Pass A를 10~15분 창 분할 + 개수 길이 비례 (09문서 1-5 확장) |
| 참가자 이름 오인식 ("재호", "지혁") | 도메인 어휘 부재 | 레슨 등록 시 참가자 이름 → initial_prompt 주입 |
| 1:1 아닌 게임 레슨 | 프롬프트에 "여성 코치→남성 수강생 1:1" 하드코딩 (8곳) | 09문서 1-4 (화자 가정 제거) — 백로그가 아니라 실손해 발생 중 |
| 깨진 인용의 노출 방식 | 인용을 LLM이 "다듬으면" 할루시네이션 재유입 | 원문 유지 + STT 신뢰도 낮으면 "발음 불명확" 표시 + 해석 라인은 "AI 해석" 라벨 (09문서 1-6) |

이 영상은 정답지를 만들 수 있으므로 **골든셋 3호**로 등록 권장.

## 4. 로드맵 — 단기/중기/장기

### 단기 (지금 스프린트): 품질 기반 다지기 — 전부 무료

1. **골든셋 회귀 스크립트** (09문서 1-3) — 이후 모든 변경의 심판. 최우선.
2. **전사 상향 3종**: initial_prompt(1-1) + 전처리(1-2) + 화자 하드코딩 제거(1-4 Phase A)
3. **프로파일 3종 구현** (10문서): `local-max` / `server-free` / `server-paid`
4. **추출 창 분할 + 개수 길이 비례** (3절 진단 반영, 09문서 1-5 확장)
5. 운영은 당분간 **local-max로 도그푸딩** (로컬 yt-dlp 정상 + 품질 최고 + 0원)

### 중기 (서비스 오픈 준비): 서버 가동 방식 선택

둘 중 하나를 선택 (병행 가능 — 10문서 폴백 체인):

- **무료 노선**: 로컬 worker 구현 (08문서 b안) — Cloud Run은 접수/조회만,
  집 머신이 Supabase 잡 폴링 → 다운로드·분석. 집 머신 가동 시간 의존이 단점.
- **유료 노선**: residential proxy (`YTDLP_PROXY`) + Groq — 회당 ~130원.
  설정만으로 오늘 가동 가능. Cloud Run `--no-cpu-throttling` 확인 필수 (10문서 5절).

+ 2계층 리포트 UI (검증 인용 vs AI 노트, 간이분석 배지) — 09문서 1-6, 10문서 3-3.

### 장기 (스케일업): Phase 3 직접 업로드 = 구조적 정답

**봇 감지·프록시 비용·yt-dlp 유지보수를 모두 소멸시키는 유일한 경로.**

- 브라우저 FFmpeg.wasm으로 오디오만 추출(1GB 영상 → ~10MB) → 서버 전송
- 비용 구조가 원천적으로 0원인 이유:
  - ingress(수신)는 모든 클라우드 무료 / egress는 JSON 텍스트뿐
  - **무저장(In-Memory)** 설계 — 오디오는 RAM에서 STT로 보내고 즉시 휘발, DB엔 텍스트만
  - STT는 Groq 무료 티어, 구조화는 Gemini 무료 티어
  - 프록시 불필요 (봇 감지 개념 소멸)
- 제약과 해법:
  - Cloud Run 요청 32MB 한도 → 오디오 압축(1시간≈14MB)으로 통과, 초과분만 분할
  - 영상 재생 UX → 유저가 로컬 파일 재선택 시 `URL.createObjectURL`로 무저장
    로컬 재생 + 타임스탬프 점프. 유튜브에도 올린 유저는 링크 병기
  - Phase 2 관절 분석은 영상 프레임 필요 → 유료 티어에서만 영상 업로드 수용
- **전략적 포지셔닝**: 유튜브 링크 경로는 "이미 유튜브에 올리는 유저용 편의
  기능"으로 격하하고, 직접 업로드를 기본 동선으로. 무료 티어가 지속 가능한
  원가 구조는 이 길뿐이다.

### 리텐션/수익 기능 (로드맵과 병행, 09문서 2절)

우선 착수: 연습 체크리스트(#2)·하이라이트 클립 공유(#4) → 유료화 시점에
고질병 트래커(#1) → B2B 씨앗으로 코치 확인 링크(#5).

## 5. 비용 전략 총괄

| 시나리오 | 오디오 획득 | STT | 회당 비용 | 봇 감지 | 비고 |
|---|---|---|---|---|---|
| 로컬 도그푸딩 (local-max) | yt-dlp 직접 | 로컬 large-v3 | 0원 | 없음 (주거 IP) | 품질 기준선 |
| 서버 무료 (server-free) | yt-dlp 시도→폴백 | Groq free tier | 0원 | **미해결** (성공률 낮음) | 폴백 품질 저하 감수 |
| 서버 유료 (server-paid) | residential proxy | Groq | ~130원 | 해결 (85~95%) | 오늘 가동 가능 |
| 로컬 worker | 집 머신 yt-dlp | 로컬/Groq | 0원 | 해결 (집 IP) | 구현 필요, 가동시간 의존 |
| **Phase 3 직접 업로드** | **획득 자체가 없음** | Groq free tier | **0원** | **개념 소멸** | **장기 정답** |
