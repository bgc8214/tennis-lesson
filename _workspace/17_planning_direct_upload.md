# 17. 직접 업로드 실행 기획 — 유튜브 링크 없이 코칭 분석 (Phase 3 구체화)

작성: 오케스트레이터 세션 / 2026-07-20
성격: **실행 기획 (구현 없음)** — 전략 차원 언급(PMD 5.3, 11문서 4절 장기,
12문서 Stage 0-2 법적 리스크, 16문서 V-2)을 한 장의 실행 스펙으로 구체화
전제: 기존 whisper 검증 파이프라인·모먼트 내비게이션을 **그대로 재사용** —
바뀌는 것은 "오디오를 어떻게 얻는가"뿐

## 0. 왜 지금 문서화하는가

직접 업로드는 세 문제를 동시에 해소하는 상용화 전제조건이다 (기존 문서 결론 재확인):
1. **봇 감지 소멸** — YouTube에서 받는 행위 자체가 없어짐 (11문서 인사이트 ③)
2. **법적 리스크 해소** — 서버 yt-dlp 다운로드의 YouTube TOS 문제 (12문서 0-2)
3. **비용 0원 구조** — ingress 무료 + 무저장 + Groq/Gemini 무료 티어 (12문서)

유튜브 링크 경로는 폐기하지 않고 "이미 유튜브에 올리는 유저용 편의 기능"으로
병행 유지한다 (11문서 포지셔닝 그대로).

## 1. UX 플로우

```
[업로드 화면] 파일 선택(mp4/mov)
  → 브라우저에서 FFmpeg.wasm으로 오디오 트랙만 추출
  → 추출 오디오(시간당 ~14MB) 업로드 + 메타(제목·길이·파일 해시)
  → 202 접수, 기존과 동일한 진행 표시(PENDING→PROCESSING→DONE)
[리포트 화면] 기존 모먼트 내비게이션 그대로
  → 단, 영상 재생은 유저가 같은 파일을 재선택 → 로컬 재생 + 타임스탬프 점프
```

핵심 UX 원칙: **영상은 서버로 가지 않는다.** 오디오만 추출·전송하고,
재생은 로컬 파일로 한다. "영상이 폰을 떠나지 않아요"를 화면에 명시 —
초상권 우려 해소이자 마케팅 카피.

## 2. 기술 스펙

### 2-1. 클라이언트 (FFmpeg.wasm)

- **오디오 추출은 재인코딩 없이 demux 우선**: `-vn -acodec copy` (mp4의 AAC
  트랙을 그대로 뽑음) — 영상 인코딩과 달리 demux는 폰에서도 초~수십 초.
  copy 불가 코덱만 aac/mp3 재인코딩 폴백. (PMD의 "0.5초" 표현은 과장 —
  실측으로 확정할 것, U-0 참조)
- 대용량 파일(1GB+) 메모리: ffmpeg.wasm의 파일 시스템 마운트/청크 처리
  지원 범위를 U-0에서 실측. 실패 시 폴백: MediaRecorder로 `<video>` 재생
  캡처(느림) 대신 **길이 제한 안내**가 정직한 대응.
- 메타 추출: duration은 `<video>` 엘리먼트, 제목은 파일명 기본값 + 수정 가능,
  썸네일은 캔버스 캡처 1장(선택, 수십 KB)만 업로드.

### 2-2. 서버 (기존 파이프라인 재사용 — 신규는 입구뿐)

- 신규 엔드포인트: `POST /lessons/analyze-upload` (multipart: 오디오 + 메타).
  Cloud Run 요청 한도 32MiB — 16kHz mono 32kbps 기준 90분 ≈ 21MB로 단일
  요청 통과. 초과분(2시간+)만 분할 업로드 or 길이 제한.
- 처리: 받은 오디오를 임시 파일 → `stt_providers.transcribe_audio()` 이후
  기존 whisper 검증 파이프라인 그대로 (`generate_lesson_report_whisper`의
  yt-dlp 다운로드 단계만 대체). **디스크/스토리지 영구 저장 없음** —
  분석 후 즉시 삭제, DB엔 텍스트 리포트만 (PMD 5.3 무저장 원칙).
- STT는 `STT_PROVIDER=groq` 권장 (서버 CPU 부담 없음, 10문서).

### 2-3. 데이터 모델

- `lessons.source_type: "youtube" | "upload"` 추가.
  upload면 `youtube_url`/`youtube_video_id` null 허용 — 중복 방지는
  video_id 대신 **파일 해시**(클라이언트 계산)로.
- 프론트 분기: `VideoPlayer`가 youtube iframe 전제라면 upload 레슨용
  로컬 파일 재생 컴포넌트(`URL.createObjectURL` + currentTime 점프) 신설.
  파일 미선택 상태에서도 리포트 텍스트/타임라인은 완전히 열람 가능해야 함
  (영상은 부가물, 리포트가 본체).

### 2-4. 비전 트랙과의 접점 (16문서 V-2)

이 문서의 U-1은 **오디오만** 다룬다. 영상 프레임이 필요한 비전(스윙 구분·
하이라이트)은 16문서 V-2의 판정(온디바이스 vs 직접 영상 업로드+서버리스 GPU)
을 따르며, 후자로 결정되면 이 문서의 업로드 플로우에 "영상 업로드(유료/
Wi-Fi 전제)" 트랙이 추가되는 구조 — 지금은 스코프 밖.

## 3. 단계별 계획

| 단계 | 내용 | 게이트 |
|---|---|---|
| **U-0. PoC 실측** | 폰 브라우저(안드/iOS Safari)에서 FFmpeg.wasm demux 실측 — 1GB/60분 파일 추출 소요 시간·메모리·실패율 | 폰에서 1분 이내 추출 성공률 — 미달 시 데스크톱 우선 출시로 조정 |
| **U-1. MVP** | 업로드 엔드포인트 + 클라이언트 추출/업로드 + source_type 분기 + 로컬 재생 | 도그푸딩: 내 레슨 영상을 유튜브 업로드 없이 분석 완료 |
| **U-2. 다듬기** | 파일 해시 중복 방지, 길이 제한 안내, 진행률 UX, 촬영 가이드 통합 | 베타 유저 투입 가능 |
| U-3 (스코프 밖) | 영상 업로드 트랙 (비전 V-2 판정 후) | — |

**우선순위**: 12문서 Stage 0 체크리스트 6번("Phase 3 개발 착수 — 베타와 병행,
상용화 전제조건")과 동일 — 베타 최소 세트(👍/👎 완료, 배포 안정화, 약관) 다음,
클로즈드 베타 기간 중 병행 개발이 적정. U-0(실측)은 반나절짜리라 지금 해도 됨.

## 4. 리스크

| 리스크 | 대응 |
|---|---|
| 폰 FFmpeg.wasm 메모리/시간 (U-0가 판정) | demux 우선 전략 + 길이 제한 + 데스크톱 우선 폴백 |
| 유저가 "왜 영상 재생이 안 돼요?" | 업로드 시·리포트 화면에 "영상은 서버에 저장되지 않아요" 명시 + 파일 재선택 UX를 원탭으로 |
| 크레딧 어뷰징 (같은 파일 반복 분석) | 파일 해시 중복 체크 (2-3) |
| 오디오 트랙 없는/무음 영상 | 기존 "전사 결과 비어 있음" 에러 경로 재사용 + 사전 클라이언트 검사(오디오 트랙 존재 여부) |
