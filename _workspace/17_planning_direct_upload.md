# 17. 직접 업로드 실행 기획 — 유튜브 링크 없이 코칭 분석 (Phase 3 구체화)

작성: 오케스트레이터 세션 / 2026-07-20
성격: **실행 기획 (구현 없음)** — 전략 차원 언급(PMD 5.3, 11문서 4절 장기,
12문서 Stage 0-2 법적 리스크, 16문서 V-2)을 한 장의 실행 스펙으로 구체화
전제: 기존 whisper 검증 파이프라인·모먼트 내비게이션을 **그대로 재사용** —
바뀌는 것은 "오디오를 어떻게 얻는가"뿐

## 0. 왜 지금 문서화하는가

직접 업로드는 세 문제를 동시에 해소하는 상용화 전제조건이다 (기존 문서 결론 재확인):
1. **봇 감지 소멸** — YouTube에서 받는 행위 자체가 없어짐 (11문서 인사이트 ③)
2. **법적 리스크 해소** — 서버 yt-dlp 다운로드의 YouTube TOS 문제 (12문서 0-2)
3. **비용 0원 구조** — ingress 무료 + 무저장 + Groq/Gemini 무료 티어 (12문서)

유튜브 링크 경로는 폐기하지 않고 "이미 유튜브에 올리는 유저용 편의 기능"으로
병행 유지한다 (11문서 포지셔닝 그대로).

## 1. UX 플로우

```
[업로드 화면] 파일 선택(mp4/mov)
  → 브라우저에서 FFmpeg.wasm으로 오디오 트랙만 추출
  → 추출 오디오(시간당 ~14MB) 업로드 + 메타(제목·길이·파일 해시)
  → 202 접수, 기존과 동일한 진행 표시(PENDING→PROCESSING→DONE)
[리포트 화면] 기존 모먼트 내비게이션 그대로
  → 단, 영상 재생은 유저가 같은 파일을 재선택 → 로컬 재생 + 타임스탬프 점프
```

핵심 UX 원칙: **영상은 서버로 가지 않는다.** 오디오만 추출·전송하고,
재생은 로컬 파일로 한다. "영상이 폰을 떠나지 않아요"를 화면에 명시 —
초상권 우려 해소이자 마케팅 카피.

## 2. 기술 스펙

### 2-1. 클라이언트 (FFmpeg.wasm)

- **오디오 추출은 재인코딩 없이 demux 우선**: `-vn -acodec copy` (mp4의 AAC
  트랙을 그대로 뽑음) — 영상 인코딩과 달리 demux는 폰에서도 초~수십 초.
  copy 불가 코덱만 aac/mp3 재인코딩 폴백. (PMD의 "0.5초" 표현은 과장 —
  실측으로 확정할 것, U-0 참조)
- 대용량 파일(1GB+) 메모리: ffmpeg.wasm의 파일 시스템 마운트/청크 처리
  지원 범위를 U-0에서 실측. 실패 시 폴백: MediaRecorder로 `<video>` 재생
  캡처(느림) 대신 **길이 제한 안내**가 정직한 대응.
- 메타 추출: duration은 `<video>` 엘리먼트, 제목은 파일명 기본값 + 수정 가능,
  썸네일은 캔버스 캡처 1장(선택, 수십 KB)만 업로드.

### 2-2. 서버 (기존 파이프라인 재사용 — 신규는 입구뿐)

- 신규 엔드포인트: `POST /lessons/analyze-upload` (multipart: 오디오 + 메타).
  Cloud Run 요청 한도 32MiB — 16kHz mono 32kbps 기준 90분 ≈ 21MB로 단일
  요청 통과. 초과분(2시간+)만 분할 업로드 or 길이 제한.
- 처리: 받은 오디오를 임시 파일 → `stt_providers.transcribe_audio()` 이후
  기존 whisper 검증 파이프라인 그대로 (`generate_lesson_report_whisper`의
  yt-dlp 다운로드 단계만 대체). **디스크/스토리지 영구 저장 없음** —
  분석 후 즉시 삭제, DB엔 텍스트 리포트만 (PMD 5.3 무저장 원칙).
- STT는 `STT_PROVIDER=groq` 권장 (서버 CPU 부담 없음, 10문서).

### 2-3. 데이터 모델

- `lessons.source_type: "youtube" | "upload"` 추가.
  upload면 `youtube_url`/`youtube_video_id` null 허용 — 중복 방지는
  video_id 대신 **파일 해시**(클라이언트 계산)로.
- 프론트 분기: `VideoPlayer`가 youtube iframe 전제라면 upload 레슨용
  로컬 파일 재생 컴포넌트(`URL.createObjectURL` + currentTime 점프) 신설.
  파일 미선택 상태에서도 리포트 텍스트/타임라인은 완전히 열람 가능해야 함
  (영상은 부가물, 리포트가 본체).

### 2-4. 공유 클립 (U-2.5) — "링크로 그 장면 보여주기"의 대체

직접 업로드의 대가는 영상이 호스팅되지 않아 **타인 공유(코치 링크, 카톡
하이라이트)에서 영상이 사라진다**는 것. 두 갈래로 대응:

- **(A) 유튜브 경로 재포지셔닝**: "공유 기능이 필요한 유저용 트랙"으로 명시.
  업로드 UI에 "코치님/친구와 영상까지 공유하려면 유튜브(비공개 링크 가능)로
  분석하세요" 안내 — 비용 0, 자기선택 구조.
- **(B) 클라이언트 사이드 클립 추출**: 전체 영상은 여전히 서버로 안 보내되,
  **유저가 공유 버튼을 눌러 능동 선택한 경우에만** 검증된 타임스탬프 상위
  ~3개 구간(각 15~30초)을 브라우저 FFmpeg.wasm으로 잘라(`-ss/-t -c copy`,
  demux 수준이라 폰에서도 빠름, 클립당 수 MB) 업로드 → 저장 → 공유 링크/
  코치 페이지에 임베드.
  - 저장소: **Cloudflare R2** (PMD 6.1 원지정 스택 — egress 무료가 정확히
    이 용도). 레슨당 ~15MB, 저장비 월 수 원 수준, 공유 트래픽 0원.
  - 무저장 원칙과의 정합: 예외를 "유저의 명시적 공유 행위로 선택된 짧은
    클립"으로 한정 — 전체 영상 무저장 유지, 초상권도 능동 공유라 성격이 다름.
  - 클립 삭제: 레슨 삭제 시 연쇄 삭제 + (선택) 만료 정책.

### 2-5. 비전 트랙과의 접점 (16문서 V-2)

이 문서의 U-1은 **오디오만** 다룬다. 영상 프레임이 필요한 비전(스윙 구분·
하이라이트)은 16문서 V-2의 판정(온디바이스 vs 직접 영상 업로드+서버리스 GPU)
을 따르며, 후자로 결정되면 이 문서의 업로드 플로우에 "영상 업로드(유료/
Wi-Fi 전제)" 트랙이 추가되는 구조 — 지금은 스코프 밖.

## 3. 화면 설계 — 컴포넌트별 변경 스펙 (현행 코드 확인 기준, 2026-07-20)

구현 세션이 파일 단위로 바로 착수할 수 있도록 현행 구조 대비 변경점을 명시한다.

### 3-1. `UrlInputForm.tsx` — 입력 방식 이원화

- 현행: URL 텍스트 입력 단일 경로 (`analyzeLesson({youtube_url})` 호출).
- 변경: 상단에 **입력 방식 탭/토글** `[유튜브 링크 | 영상 파일]`.
  - 유튜브 탭: 현행 그대로.
  - 파일 탭: 파일 선택(`accept="video/*"`) → 상태 머신
    `선택됨 → 오디오 추출 중(진행률) → 업로드 중(진행률) → 접수(202)`.
    추출은 FFmpeg.wasm demux(2-1), 실패 시 명확한 에러("이 형식은 지원하지
    않아요")와 유튜브 경로 안내.
  - 파일 탭에 안내 문구 2개 고정: "영상은 서버로 전송되지 않고, 소리만
    분석에 사용돼요" / "코치님·친구와 영상까지 공유하려면 유튜브 링크로
    분석하세요" (2-4 A안).
  - 기존 촬영 가이드 `<details>`는 두 탭 공통 유지.

### 3-2. `VideoPlayer.tsx` — 소스 분기 (핵심 변경)

- 현행: props가 `youtubeUrl`/`youtubeVideoId` 필수, YT iframe API 하드코딩.
- 변경: `ReportView/index.tsx`에서 `lesson.source_type`으로 분기.
  - `youtube`: 현행 `VideoPlayer` 그대로.
  - `upload`: 신규 `LocalVideoPlayer` 컴포넌트:
    - 파일 미선택 상태(기본): placeholder 카드 — "영상 파일 선택" 버튼 +
      "영상은 서버에 저장되지 않아 기기에서 다시 선택해요" 문구.
      **리포트 텍스트/타임라인은 파일 없이도 완전 열람 가능** (영상은 부가물).
    - 선택 후: `URL.createObjectURL`로 `<video>` 재생. `requestedSec` prop을
      받아 `currentTime` 점프 — 기존 `clampToDuration` 로직 이식
      (`video.duration` 기준). 언마운트 시 `revokeObjectURL`.
    - (선택) 파일 해시를 레슨의 `file_hash`와 대조해 "다른 파일 같아요" 경고.
- `handleSeek` 흐름(index.tsx의 `requestedSec` state)은 두 플레이어 공용 —
  변경 불필요.

### 3-3. `ReportView/index.tsx`, 타입, API

- `LessonDetail`에 `source_type`, `file_hash` 추가 (`types/lesson.ts`,
  `api.ts` 매핑 — **주의: api.ts는 Supabase 직접 조회 경로가 있어 매핑 누락
  버그 전례 있음**(5ba97fc에서 발견), 백엔드 응답과 양쪽 다 반영).
- `youtube_url`/`youtube_video_id` nullable 처리에 따른 컴파일 에러 지점
  일괄 정리 (LessonCard 썸네일, ShareButtons 링크 생성부 등).

### 3-4. `LessonCard.tsx` — 목록 표시

- upload 레슨: 유튜브 썸네일 URL이 없으므로 클라이언트 캡처 썸네일(2-1,
  업로드 시 캔버스 1장) 또는 기본 플레이스홀더 + "📁 업로드" 뱃지.

### 3-5. `ShareButtons.tsx` + `/share/[token]` — 공유 분기

- youtube 레슨: 현행 유지 (타임스탬프 링크 포함).
- upload 레슨:
  - 텍스트 복사/코치 링크는 그대로 동작 (텍스트 기반).
  - "하이라이트 클립 포함" 옵션 → 2-4(B) 클립 추출 흐름. 클립 업로드 전
    동의 문구 필수: "선택한 15초 구간 N개가 공유용으로 저장됩니다".
  - `/share/[token]` 코치 페이지: `clips[]` 있으면 클립 플레이어 임베드,
    없으면 현행 텍스트 전용 + "영상은 학생에게 요청하세요" 안내.

### 3-6. 백엔드 계약 요약

```
POST /lessons/analyze-upload   multipart(audio, title?, duration_sec,
                               file_hash, thumbnail?)  → 202 {lesson_id}
POST /lessons/{id}/clips       multipart(clip, sec)     → {clip_url}  (U-2.5)
lessons: + source_type text default 'youtube', + file_hash text nullable,
         youtube_url/youtube_video_id nullable로 완화
clips 저장: R2 (또는 임시로 Supabase Storage — U-2.5에서 결정)
```

## 4. 단계별 계획

| 단계 | 내용 | 게이트 |
|---|---|---|
| **U-0. PoC 실측** | 폰 브라우저(안드/iOS Safari)에서 FFmpeg.wasm demux 실측 — 1GB/60분 파일 추출 소요 시간·메모리·실패율 | 폰에서 1분 이내 추출 성공률 — 미달 시 데스크톱 우선 출시로 조정 |
| **U-1. MVP** | 업로드 엔드포인트 + 클라이언트 추출/업로드 + source_type 분기 + 로컬 재생 | 도그푸딩: 내 레슨 영상을 유튜브 업로드 없이 분석 완료 |
| **U-2. 다듬기** | 파일 해시 중복 방지, 길이 제한 안내, 진행률 UX, 촬영 가이드 통합 | 베타 유저 투입 가능 |
| **U-2.5. 공유 클립** | 2-4(B) 클립 추출·R2 저장·공유/코치 페이지 임베드 | 클립 공유율로 수요 확인 |
| U-3 (스코프 밖) | 영상 업로드 트랙 (비전 V-2 판정 후) | — |

**우선순위**: 12문서 Stage 0 체크리스트 6번("Phase 3 개발 착수 — 베타와 병행,
상용화 전제조건")과 동일 — 베타 최소 세트(👍/👎 완료, 배포 안정화, 약관) 다음,
클로즈드 베타 기간 중 병행 개발이 적정. U-0(실측)은 반나절짜리라 지금 해도 됨.
